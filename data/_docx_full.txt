1. Project Overview & System Vision

BuzzNa D74 is engineered as an offline-first intelligent business operating system specifically tailored for micro, small, and medium enterprises (MSMEs) operating in sub-Saharan Africa and analogous emerging markets. Traditional SaaS and POS platforms fundamentally fail these businesses because they operate on flawed assumptions: constant cloud connectivity, stable municipal electricity, high formal digital literacy, and expensive monthly subscriptions.

BuzzNa D74 completely upends this paradigm by serving as a comprehensive business memory and action engine, a low-cost digital control center, and a proactive shop assistant. The primary architectural milestone of BuzzNa D74 is to minimize administrative overhead to achieve a state of 'zero attendance' operations, meaning cashiers, attendants, and business owners spend near-zero effort operating the software.

1.1 Target Business Segments & Onboarding Tailoring

Retail Shops & Wholesalers: Requires high-velocity barcode scanning, bulk break price tiers, variable dimensions, and swift payment-to-sale matching.

Butcheries: Demands precise decimal fractional units (grams up to three decimal indices), perishable batch tracking, historical buying cost variation logs, and automated spoilage account routing.

Mitumba (Apparel): Requires itemized identity stock tags, haggle margin validation guards, open-bale cost-averaging, and real-time dispatch alerts.

Hardware Stores & Agrovets: Strict serialization cross-referencing, manufacturer catalogs, legal compliance expiration trackers, low-stock threshold triggers, and extensive supplier credit frameworks.

Cyber Cafes: Time-slot calendar reservations, hourly service fee algorithms, non-physical resource indices, and recurring utility bill cost trackers.

2. Architectural Principles & Technology Stack

To guarantee enterprise-grade data integrity and production safety under challenging operational environments, the platform strictly enforces the following core architectural rules:

Offline-First & Local-Authority: Terminals must perform local validation lookups, write data states locally via IndexedDB (Dexie), and queue synchronizations asynchronously without blocking active checkouts.

Authoritative Ledger Sourcing: Inventory integer/decimal fields must never be manipulated directly via update queries. Stock levels are calculated dynamically via sequential aggregation over immutable tracking rows.

Absolute Separation of Payment Domains: Platform billing fees (via Paystack) must never mix with client merchant operational transactions (via Daraja M-Pesa).

History Preservation: Financial rows and event streams are strictly append-only. Deleting records is strictly banned. Error correction relies exclusively on compensating entries like voids, refunds, and reversals.

Strict Tenant Isolation: Every database statement, cached value, and background execution must evaluate row permissions based on the user's bound token.

Decimal Integrity: All monetary amounts must use exact decimal/numeric types (NUMERIC(15,2) or NUMERIC(12,2)), never floating point.

2.1 Technology Stack

Frontend Client Layer: React + Vite PWA + TypeScript + TanStack Query + React Router. Local offline storage powered by IndexedDB via Dexie.

Backend API Layer: Node.js + TypeScript + Express.js modular monolith.

Database Engine: PostgreSQL 16+ using Drizzle ORM or Kysely for type-safe compile-time queries, utilizing native PostgreSQL Row-Level Security (RLS).

Asynchronous Queues: Redis + BullMQ handles asynchronous background jobs, transactional retries, and report compilation.

External Integrations: Paystack API for platform subscription handling and Safaricom Daraja M-Pesa API for client-facing retail revenue collection.

4. Multi-Tenant Security & Role-Based Access (RBAC)

BuzzNa D74 implements a cost-efficient single-database, shared-schema multi-tenant model protected by a comprehensive dual-layer security framework.

4.1 Layer 1: Middleware Context Enforcement

Incoming HTTP requests pass through isolated middleware that unpacks the Bearer JWT, validates the cryptographic signature using the JWT_ACCESS_SECRET, and extracts the tenant_id UUID from payload claims. It attaches this value directly to the internal request context, explicitly ignoring client-supplied headers (e.g., X-Tenant-ID) to neutralize tampering vulnerabilities.

4.2 Layer 2: Database Row-Level Security & Connection Pool Integrity

PostgreSQL Row-Level Security (RLS) policies are activated across all tables. Because connections are reused in connection poolers (like PgBouncer), every database query MUST wrap the tenant scoping command and the actual database operation inside a strict transaction block:

BEGIN;SET LOCAL app.current_tenant_id = 'uuid_here';-- Execute real database query statements hereCOMMIT;

4.4 Shift Accountability & Handover Rules

Single Active Session: A Cashier can only have exactly one open till session active simultaneously.

Blind Balance Discrepancy Gate: At shift handover, the system presents a blind entry text input for the physical cash balance. The expected balance (Opening Float + Cash Revenue - Cash Outflows) is hidden from the cashier.

Lockout: If the physical entry variance exceeds limits, the session is flagged as REVIEW_REQUIRED, instantly locking checkout routes for that cashier and raising a high-priority attention card.

7. Core Business Logic & State Machines

7.1 Authoritative Event-Sourced Inventory Engine

BuzzNa D74 strictly prohibits direct alterations to current stock levels. The system utilizes a pure event-sourced stock valuation engine to maintain absolute historical integrity.

The Ground Truth Ledger: The authoritative source for all stock availability is the immutable inventory_events table. Every operational action (restock, sale dispatch, physical count) appends a new event row. The current_quantity field in the products table is strictly treated as a cached projection for UI rendering.

7.2 The Master Refund & Inventory Restoration Recipe

To preserve audit history, direct deletion of sales rows is strictly prohibited. Error correction follows this sequence:

Verify Tenant Boundary: Load the target transaction record and assert it matches the authenticated context.

State Duplication Check: Ensure it has not already been fully refunded or voided.

Refund Record Insertion: Insert an un-deletable tracking row into the sale_refunds table.

Inventory Restoration: If physical products are returned, append a new tracking row to inventory_events categorized as REFUND_RETURN with a positive quantity delta.

Ledger Reconciliation: Adjust customer debt books (if applicable) and write to the system security ledger.

Terminals operate at zero-latency against IndexedDB cache stores powered by Dexie.js during internet blackouts.

8.2 The "Walkaway" Sync Conflict Protocol

If a terminal syncs an offline cash sale resulting in negative inventory (e.g., another terminal depleted the stock during a network dropout), the server MUST NEVER reject the cash transaction. Rejecting it breaks physical cash drawer balance since the goods have already walked away with the customer. Instead, the server accepts the sale, drops stock to negative integers, and flags an 'Inventory Anomaly' attention card.

8.3 Offline M-Pesa Paradox Mitigation

Safaricom Daraja requires cloud webhooks. To support offline checkouts seamlessly, an Android background service intercepts local 'MPESA' SMS receipts, parsing them temporarily into IndexedDB to validate payments locally before eventual cloud sync and authoritative reconciliation.

10. Security Audit, Bugs & Vulnerabilities Analysis

A comprehensive source code and architectural audit highlights critical engineering rules necessary to prevent severe vulnerabilities:

Connection Pool Leakage Prevention: In backends using PgBouncer, failing to wrap connection settings inside an explicit transaction block allows reused pool sockets to retain the previous tenant's ID context. Fix: Mandate strict `BEGIN; SET LOCAL app.current_tenant_id... COMMIT;` transaction encapsulation for every query.

Device Clock Tampering: Malicious operators can change mobile device clocks backward to forge historical sales records and bypass subscription expiration dates. Fix: The server evaluates event timestamps against server suspension logs and rejects back-dated writes into the `sync_rejections` queue.

Zero Client-Side Trust: Never trust the client browser for permission checks, tenant scope, or calculation totals. Financial rows and event streams must be re-validated on the backend monolith using server-side ledger states.

Idempotency Guards: Webhooks utilize provider tracking tokens as idempotency keys. This prevents double-crediting a tenant's account during network retries or Paystack/Daraja redeliveries.

API Rate Limiting: Strict token-bucket algorithms on /auth (5 req/min) to prevent brute force, and /sync (10 req/5min) to prevent offline clients from getting stuck in synchronization loops that DDoS the central server.

11. Deployment Strategy & Infrastructure Guide

To realize the highly scalable architecture of BuzzNa D74, deployment of the decoupled systems must follow explicit environment configurations:

11.1 Database Architecture (Neon PostgreSQL)

The shared-schema multi-tenant relational database is heavily reliant on connection pooling to handle massive offline-sync batch requests. This infrastructure maps perfectly to Neon PostgreSQL. Neon's serverless edge computing and native connection pooling ensures that the mandatory tenant context injection executes seamlessly without exhausting database sockets during peak retail hours.

11.2 Backend Monolith Pipeline (GitHub to Render)

The Node.js API monolith should be deployed directly from the GitHub repository onto Render. Render's automated build pipelines instantly compile the TypeScript source, while its native support for Background Workers provides an ideal hosting environment for the BullMQ/Redis async processors.

11.3 Android APK Preparation (Capacitor/Cordova)

To transition the React PWA into a standalone Android APK suitable for low-end mobile devices in open-air markets, the system utilizes a hybrid container architecture:

Webview Wrapper: Capacitor.js or Apache Cordova wraps the compiled React/Vite build folder into a native Android Studio project.

Native Android Service: A background Java/Kotlin service worker is embedded into the APK to register a BroadcastReceiver. This receiver listens for incoming system SMS alerts to intercept and parse 'MPESA' sender codes locally.

Peripheral Interfacing: Hardware integration layers use Capacitor-Community plugins to establish serial communication with bluetooth thermal printers and embedded hardware barcode scanners.

5. Business Rules & Edge Cases

Authoritative Ledger Sourcing: Inventory integer/decimal fields MUST NEVER be manipulated directly via update queries. Stock is calculated dynamically via sequential aggregation over immutable tracking rows.

Walkaway Sync Protocol: If an offline terminal syncs a cash sale resulting in negative inventory, the server MUST NEVER reject the cash transaction. It drops stock to negative integers and flags an Attention Card.

Blind Handover Discrepancy Gate: Cashiers cannot see expected till balance during closure. If physical cash entry variance exceeds limits, the session locks (REVIEW_REQUIRED).

History Preservation & Refunds: Direct deletion of financial rows is banned. Error correction relies exclusively on voids, refunds, and appending REFUND_RETURN inventory events.

License Enforcement: PAYMENT_DUE retains a 3-day grace period. SUSPENDED_NON_PAYMENT enforces a strict read-only lock, blocking all POS writes and syncing.

Decimal Integrity: All monetary amounts must strictly use NUMERIC(15,2) or NUMERIC(12,2) to avoid floating-point rounding errors.
